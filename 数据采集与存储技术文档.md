# 物联网数据采集与存储系统技术文档

## 一、数据流转概述

本系统采用MQTT协议进行物联网设备数据采集，数据流转流程如下：

1. **数据采集**：设备数据通过MQTT协议发布到指定主题，系统作为MQTT客户端订阅相关主题接收数据。
2. **数据接收**：系统通过`on_message`函数接收MQTT消息，根据不同的消息协议类型选择相应的处理函数。
3. **数据解析与转换**：`handle_energy_message_cetc.py`中的`handle`函数负责解析原始数据，进行必要的计算和转换。
4. **数据缓存**：处理后的数据先存储到Redis中，用于设备状态更新和短期缓存。
5. **数据持久化**：最终将数据插入到MySQL数据库中，同时根据需要写入默认表和月度表。

系统支持多种能源类型数据采集，包括电力、天然气、水等，并具有数据校验、过滤和告警处理功能。

### 3. 去重与数据校验机制

系统实现了多级数据校验和过滤机制，确保数据质量：

1. **设备有效性校验**：检查设备是否在已知设备列表中

#### 3.1 数据完整性校验

对关键电气参数(uab, ubc, uac, pw等)进行求和验证，若总和小于等于0，则判定为无效数据并丢弃。

校验公式：
```
sum_val = uab + ubc + uac + pw + ua + ub + uc + tp + pr + st + fr
if sum_val <= 0:
    丢弃数据
```

代码实现：
```python
# 判断是否读到的数据存不存在 如果字段不存在 那么认为该设备是不在线/异常的
if (
    not current_config.INSERT_NONE_COLLECT_LOG
    and energy_type == EnergyType.ELECTRIC_ENERGY
):
    sum_val = sum(
        [
            data.get("uab", 0) or 0,
            data.get("ubc", 0) or 0,
            data.get("uac", 0) or 0,
            data.get("pw", 0) or 0,
            data.get("ua", 0) or 0,
            data.get("ub", 0) or 0,
            data.get("uc", 0) or 0,
            # 传感器设备
            data.get("tp", 0) or 0,
            data.get("pr", 0) or 0,
            data.get("st", 0) or 0,
            data.get("fr", 0) or 0,
        ]
    )
    if sum_val <= 0:
        logger.warning(
            "字段uab ubc uac pw sum <= 0; sum = "
            + str(sum_val)
            + " collect_device_id: "
            + str(data.get("collect_device_id"))
        )
        continue
```

#### 3.3 重复数据处理

对电能数据实现特殊处理逻辑，防止因设备重启等原因导致的数据回退。

处理逻辑：
1. 记录设备上次的电能值(last_energy_usage)
2. 当前电能值(pw)如果小于上次记录值，则可能是设备重启导致的数据回退
3. 允许最多3次连续的回退情况，超过则丢弃数据
4. 正常情况下更新设备的电能值记录

代码实现：
```python
global g_device_energy
global g_device_energy_fail_count
collect_device_id = data.get("collect_device_id")
last_energy_usage = g_device_energy.get(collect_device_id) or 0
try:
    if (
        energy_type == EnergyType.ELECTRIC_ENERGY
        and current_config.PRODUCT_NAME == XiaoMiConfig.PRODUCT_NAME
    ):
        pw = int(data.get("pw", 0) or 0)

        if last_energy_usage:
            if pw and pw < last_energy_usage:
                fail_count = (
                    g_device_energy_fail_count.get(collect_device_id) or 0
                )
                fail_count += 1
                if fail_count <= 3:
                    logger.info(
                        f"此设备{data.get('collect_device_id')} 上次pw {last_energy_usage} 本次pw {pw} 丢弃"
                    )
                    g_device_energy_fail_count[collect_device_id] = fail_count
                    continue
        if pw:
            g_device_energy[collect_device_id] = pw

except Exception as e:
    logger.info(f"error 0x0334 {e}")
```

```python
def _process_prev_last_data(data):
    cur_t: datetime.datetime = data["collect_time"]
    cur_t = cur_t - datetime.timedelta(hours=1)
    insert_time = cur_t.replace(minute=59, second=59, microsecond=99)
    collect_device_id = data["collect_device_id"]
    logger.debug(
        f"插入数据 {insert_time} collect_device_id {collect_device_id} cur_t{cur_t}"

    if cur_t.hour != last_t.hour or cur_t.day != last_t.day:
        data["collect_time"] = insert_time
        _update_device_last_insert_hour(collect_device_id, cur_t)
        _insert_into_table(data, ENERGY_TABLE_NAME_DEFAULT())
```

## 五、/history主题数据处理流程

### 1. 数据解析流程

当系统接收到发布到/history主题的数据时，会按照以下流程进行解析：

1. **消息接收**：MQTT客户端通过`on_message`函数接收来自设备的/history主题消息
2. **主题识别**：系统根据配置文件中的`MQTT_TOPICS`映射关系识别消息对应的能源类型
3. **数据解码**：将JSON格式的消息体解析为Python字典对象
4. **设备数据提取**：遍历消息中的设备列表(`devs`)，对每个设备进行单独处理
5. **字段映射**：通过`_parse_single_device_data`函数将原始测量点名称映射到数据库字段
6. **数据转换**：根据设备配置的变比(TR)对原始数据进行计算转换

### 2. 数据校验机制

/history主题的数据在校验方面采用了多层次的验证机制：

1. **设备合法性校验**：
   - 检查设备名称是否存在于系统配置的设备列表中
   - 验证设备ID映射是否正确

2. **数据完整性校验**：
   - 对关键电气参数(uab, ubc, uac, pw等)进行求和验证
   - 若总和小于等于0，则判定为无效数据并丢弃

3. **数据范围校验**：
   - 检查电能数据(pw)是否超过预设的最大值阈值
   - 验证气体/水表数据(st)是否低于最小值限制

4. **重复数据处理**：
   - 对电能数据实现特殊处理逻辑，防止因设备重启等原因导致的数据回退

### 3. 数据持久化流程

/history主题数据的持久化过程包括以下几个步骤：

1. **实时状态更新**：
   - 将处理后的数据存储到Redis缓存中，用于设备状态展示
   - 更新设备在线状态标识

2. **数据库写入**：
   - 使用事务机制将数据同时写入两个表：
     - 默认表`collect_energy`：存储最新的数据记录
     - 月度分表`collect_energy_YYYY_MM`：按月份分区存储历史数据

3. **历史数据补录**：
   - 通过`_process_prev_last_data`函数处理上一个小时的数据完整性
   - 确保每小时都有完整的数据记录用于后续统计分析

4. **小时统计数据生成**：
   - 系统定时任务(job/statistics_job.py)定期执行小时数据统计
   - 从`collect_energy`表中查询数据，计算最大值、最小值等统计指标
   - 将统计结果写入`stat_hour`表，用于报表展示和分析

### 4. 数据处理特点

1. **全量数据处理**：与/rtg主题的实时数据不同，/history主题的数据会经过完整的处理流程，包括持久化到MySQL数据库
2. **多重数据校验**：确保入库数据的质量和准确性
3. **时间序列存储**：支持按时间维度的数据查询和分析
4. **统计分析支持**：为上层应用提供小时级别的统计数据
